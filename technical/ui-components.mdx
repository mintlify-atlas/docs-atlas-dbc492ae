## Overview

ThystTV's UI is built with modern Android components: Fragments, ViewBinding, ViewModel, and Jetpack Navigation. The app follows single-activity architecture with Fragment-based navigation.

## Core Fragments

### PlayerFragment

The video/stream player with advanced controls and PiP support.

**Location**: `ui/player/PlayerFragment.kt`

```kotlin
@AndroidEntryPoint
class PlayerFragment : BaseNetworkFragment() {
    private var _binding: FragmentPlayerBinding? = null
    private val binding get() = _binding!!
    private val viewModel: PlayerViewModel by viewModels()
}
```

<Accordion title="PlayerFragment Features">
  - **Multi-format playback**: Supports streams, VODs, and clips
  - **Picture-in-Picture**: Android PiP mode support
  - **Advanced controls**: Quality selection, speed control, sleep timer
  - **Gesture controls**: Swipe for volume/brightness, double-tap to seek
  - **Chat integration**: Side-by-side chat view
  - **Stats tracking**: Records watch sessions for analytics
  - **Offline support**: Plays downloaded videos
</Accordion>

### StatsFragment

Viewing statistics and analytics dashboard.

**Location**: `ui/stats/StatsFragment.kt`

```kotlin

    private val viewModel: StatsViewModel by viewModels()
    private var binding: FragmentStatsBinding? = null
}
```

<Accordion title="StatsFragment Displays">
  - **Screen time chart**: 7-day bar chart with daily totals
  - **Watch streaks**: Current and longest viewing streaks
  - **Top channels**: Most-watched streamers with total time
  - **Category breakdown**: Pie chart of game categories
  - **Peak times heatmap**: Hour-of-day viewing patterns
  - **Loyalty metrics**: Consecutive days per streamer
</Accordion>

#### Stats Data Flow

```kotlin
override fun onViewCreated(view: View, savedInstanceState: Bundle?) {
    viewLifecycleOwner.lifecycleScope.launch {
        viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
            launch {
                viewModel.screenTime.collectLatest { screenTimes ->
                    updateScreenTimeDisplay(binding, screenTimes)
                }
            }
            launch {
                viewModel.topStreams.collectLatest { streams ->
                    adapter.submitList(streams)
                }
            }
            launch {
                viewModel.watchStreak.collectLatest { streak ->
                    updateStreakDisplay(binding, streak)
                }
            }
        }
    }
}
```

<Expandable title="Why multiple launch blocks?">
  Each `launch` collects from a separate Flow concurrently. This allows different parts of the UI to update independently as new data arrives, improving responsiveness.
</Expandable>

## ViewBinding Pattern

All fragments use ViewBinding for type-safe view access:

```kotlin
private var _binding: FragmentExampleBinding? = null
private val binding get() = _binding!!

override fun onCreateView(
    inflater: LayoutInflater,
    container: ViewGroup?,
    savedInstanceState: Bundle?
): View {
    _binding = FragmentExampleBinding.inflate(inflater, container, false)
    return binding.root
}

override fun onDestroyView() {
    super.onDestroyView()
    _binding = null
}
```

<Accordion title="Why nullable backing property?">
  The binding should only exist between `onCreateView` and `onDestroyView`. Setting `_binding = null` in `onDestroyView` prevents memory leaks by releasing view references when the fragment's view is destroyed.
</Accordion>

## Lifecycle-Aware Data Collection

Fragments collect Flow data in a lifecycle-safe manner:

```kotlin
viewLifecycleOwner.lifecycleScope.launch {
    viewLifecycleOwner.repeatOnLifecycle(Lifecycle.State.STARTED) {
        viewModel.uiState.collectLatest { state ->
            updateUI(state)
        }
    }
}
```

This ensures data collection starts when STARTED and stops when STOPPED, preventing wasted resources and crashes.
