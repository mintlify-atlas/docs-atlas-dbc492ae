## Overview

ThystTV implements the Repository pattern to provide a clean API for data access. Repositories abstract the data layer, handling both local database operations and network requests to Twitch APIs.

## Architecture Pattern

The repository layer follows these principles:

- **Single Source of Truth**: Repositories are the authoritative data source for ViewModels
- **Dependency Injection**: All repositories are singletons injected via Hilt
- **Coroutine-based**: All operations use Kotlin coroutines for async execution
- **Context Switching**: Network/DB operations execute on `Dispatchers.IO`

## Local Data Repositories

These repositories manage local database operations.

### BookmarksRepository

Manages saved video bookmarks with file cleanup logic.

```kotlin
@Singleton
class BookmarksRepository @Inject constructor(
    private val bookmarksDao: BookmarksDao,
    private val videosDao: VideosDao
) {
    fun loadBookmarksFlow() = bookmarksDao.getAllFlow()
    
    suspend fun loadBookmarks() = withContext(Dispatchers.IO) {
        bookmarksDao.getAll()
    }
    
    suspend fun getBookmarkByVideoId(id: String) = withContext(Dispatchers.IO) {
        bookmarksDao.getByVideoId(id)
    }
    
    suspend fun saveBookmark(item: Bookmark) = withContext(Dispatchers.IO) {
        bookmarksDao.insert(item)
    }
    
    suspend fun deleteBookmark(item: Bookmark) = withContext(Dispatchers.IO) {
        // Clean up cached thumbnails and logos
        if (!item.videoId.isNullOrBlank() && videosDao.getByVideoId(item.videoId).isEmpty()) {
            item.thumbnail?.let { File(it).delete() }
        }
        if (!item.userId.isNullOrBlank() && 
            getBookmarksByUserId(item.userId).none { it.id != item.id }) {
            item.userLogo?.let { File(it).delete() }
        }
        bookmarksDao.delete(item)
    }
}
```

<Accordion title="Why file cleanup in repository?">
  When deleting bookmarks, the repository checks if the associated video or channel has other references. If not, it deletes cached image files to free up storage. This business logic is centralized in the repository rather than scattered across ViewModels.
</Accordion>

## Network Repositories

These repositories handle API communication with Twitch services.

### GraphQLRepository

Handles all Twitch GraphQL API requests with multi-network library support.

```kotlin
@Singleton
class GraphQLRepository @Inject constructor(
    private val httpEngine: Lazy<HttpEngine>?,
    private val cronetEngine: Lazy<CronetEngine>?,
    private val cronetExecutor: ExecutorService,
    private val okHttpClient: OkHttpClient,
    private val json: Json
) {
    private suspend fun <T: Query.Data> sendQuery(
        networkLibrary: String?,
        headers: Map<String, String>,
        query: Query<T>
    ): ApolloResponse<T>
}
```

<Accordion title="Network Library Selection">
  GraphQLRepository supports three network backends:
  
  1. **HttpEngine** - Android's new HTTP client (Android S+, API 31+)
  2. **Cronet** - Google's Chromium network stack
  3. **OkHttp** - Fallback for older devices
  
  The library is selected at runtime based on user preferences and device capabilities.
</Accordion>

### PlayerRepository

Specialized repository for playback-related operations.

```kotlin
@Singleton
class PlayerRepository @Inject constructor(
    private val httpEngine: Lazy<HttpEngine>?,
    private val cronetEngine: Lazy<CronetEngine>?,
    private val cronetExecutor: ExecutorService,
    private val okHttpClient: OkHttpClient,
    private val json: Json,
    private val graphQLRepository: GraphQLRepository,
    private val helixRepository: HelixRepository,
    private val recentEmotes: RecentEmotesDao,
    private val videoPositions: VideoPositionsDao
)
```

<Accordion title="PlayerRepository Responsibilities">
  - Fetch playback access tokens
  - Load HLS playlists
  - Manage video positions for resume
  - Handle third-party emote services (BTTV, FFZ, 7TV)
  - Load chat badges and emotes
  - Fetch recent chat messages
</Accordion>

## Data Source Pattern

For paginated content, ThystTV uses custom DataSource classes:

```kotlin
class FollowedStreamsDataSource(
    private val helixClientId: String?,
    private val helixToken: String?,
    private val helixApi: HelixApi,
    private val gqlClientId: String?,
    private val gqlApi: GraphQLApi
) : PagingSource<Int, Stream>()
```

These integrate with Jetpack Paging 3 for efficient infinite scrolling.

## Repository Dependency Graph

```
ViewModel
    ↓
Repository
    ↓
    ├─→ DAO (Room Database)
    ├─→ GraphQLRepository (Twitch GQL API)
    ├─→ HelixRepository (Twitch Helix API)
    └─→ PlayerRepository (Playback & Chat)
```

## Error Handling

Repositories use `withContext(Dispatchers.IO)` for all operations and let exceptions propagate to ViewModels, which handle them appropriately for UI display.
